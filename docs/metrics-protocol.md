# Volume Lighting 指标测试协议

## 1. 目的

本协议用于统一每轮修改完成后的测试指标、测试场景、记录格式和通过标准。

所有迭代在提交代码前，必须至少完成一次指标测试并写入对应迭代记录。

## 2. 指标来源与设计依据

本项目指标设计基于以下对标论文与当前核心算法目标：

- Clustered Deferred and Forward Shading
- Forward+
- Efficient Real-Time Shading with Many Lights

当前核心算法目标：

- 在 Clustered Light List 基线之上，引入单 Volume 单等效光源表示
- 对比性能收益与视觉误差
- 评估不同场景下的稳定性、计算资源消耗、存储资源消耗与画质损失

本协议优先吸收以下适用于实时渲染论文的指标思路：

- 性能效率优先：帧时间、FPS、分阶段耗时、随光源数量增长的扩展曲线
- 图像质量定量化：PSNR、SSIM、RMSE、FLIP
- Ground Truth 对照：优先使用高质量离线或高质量 RTX 参考结果

## 3. 固定测试场景

每轮测试默认使用以下场景：

1. `Sponza`
   用于性能、遮挡复杂度和分块规模验证

2. `MetalRoughSpheres`
   用于材质与高光误差验证

3. `DamagedHelmet`
   用于法线细节和近景高光误差验证

如果某轮改动只影响场景基础设施，可只测受影响场景；如果改动影响算法或 shader，必须覆盖全部三场景。

## 4. 固定对比模式

每轮测试至少包含以下模式：

1. `ClusteredBaseline`
   当前引擎原始 Clustered Light List 路径

2. `EquivalentVolumeV1`
   当前轮次的 Volume 等效光源路径

如果当前轮次尚未引入 `EquivalentVolumeV1`，则仍需记录 baseline 指标，作为后续对照起点。

## 5. 核心指标

### 5.1 性能指标

- 平均帧时间 `ms/frame`
- 平均帧率 `FPS`
- 光照 Pass GPU 时间
- Light Injection / Voxelization 阶段耗时
- Shading 阶段耗时
- 相对加速比
  计算方式：`Baseline 帧时间 / 当前方案帧时间`
- Throughput per Light
  用于衡量单位光源对应的运行成本与扩展性
- 光源数量扩展曲线
  固定记录 `1 / 10 / 50 / 100` 个光源下的变化趋势

说明：

- 对于 `VEL` 路径，重点观察是否接近常数复杂度
- 对于 baseline，重点观察是否呈线性增长
- 若某场景暂时无法稳定容纳指定光源数量，需在记录中注明实际测试数量

### 5.2 计算资源指标

- GPU 显存占用
- 等效光源缓冲大小
- 光源索引缓冲大小
- 主要 SSBO / 纹理资源大小
- Voxel / Volume Grid 显存占用 `MB`

说明：

- 若当前轮次无法直接自动采集完整显存占用，则至少记录新增 GPU 缓冲大小的理论值或实测值
- 需要明确区分 baseline 额外资源与当前方案新增资源
- 对于轻量化结论，需明确标注 grid 分辨率与对应显存大小，例如 `32^3`

### 5.3 存储资源指标

- 场景资源目录大小
- 截图与实验记录产物大小
- 新增实验数据文件大小

说明：

- 该指标主要用于记录算法引入后的附加存储成本
- 不要求每轮重新统计全部仓库大小，但要记录本轮新增产物的体积

### 5.4 画质指标

- PSNR
- SSIM
- RMSE
- FLIP
- 高光区域误差
- 法线细节区域主观失真描述
- 光照连续性观察
- 颜色偏差观察

说明：

- 图像质量指标默认以 Ground Truth 为参考
- Ground Truth 优先采用高质量离线渲染或高质量 RTX 参考结果
- 若本地暂时不具备 `RTX 5090` 或离线参考生成条件，需先记录固定机位截图与主观误差，并在文档中标注“定量画质指标待补”

### 5.5 图像产物要求

每轮算法或 shader 轮次必须保存以下图像产物：

- 运行时图片
  反映程序成功运行、场景加载和模式切换状态

- 光照效果图
  至少包含 baseline 与当前算法路径各 1 张固定机位截图

- Ground Truth 对照图
  若当前轮次已具备高质量参考结果，需同时保存对照图

- 如存在明显失真区域，需补充局部问题截图

建议保存形式：

- `docs/artifacts/iter-001/sponza-baseline.png`
- `docs/artifacts/iter-001/sponza-equivalent-v1.png`

### 5.6 稳定性指标

- 是否成功编译
- 是否成功启动
- 是否成功切换场景
- 是否成功切换渲染模式
- 是否出现闪烁、崩溃、黑屏或明显错误光照

## 6. 每轮必测项

### 6.1 文档或流程轮次

若本轮只改文档、流程或非渲染逻辑：

- 编译验证：可选
- 启动验证：至少一次
- 记录：说明“本轮无算法指标变化”

### 6.2 场景基础设施轮次

若本轮改动场景切换、场景配置或基础运行逻辑：

- 至少测试受影响场景
- 记录启动成功与否
- 记录运行时图片是否正常

### 6.3 算法或 shader 轮次

若本轮改动算法、SSBO、聚合逻辑或 shader：

- 必测三场景
- 必测 baseline 与当前算法路径
- 必须记录性能指标
- 必须记录光源数量扩展曲线
- 必须记录计算资源指标
- 必须记录存储资源指标
- 必须记录至少 1 组运行时图片和 1 组光照效果图
- 若具备 Ground Truth，必须记录 PSNR / SSIM / RMSE / FLIP
- 必须记录主观误差结论

## 7. 固定记录字段

每轮测试记录必须包含：

- 轮次编号
- 日期
- 提交哈希
- 本轮目标
- 测试场景
- 测试模式
- 编译结果
- 启动结果
- 平均帧时间
- FPS
- 光照 Pass GPU 时间
- Light Injection / Voxelization 阶段耗时
- Shading 阶段耗时
- 光源数量测试点
- Throughput per Light
- GPU 显存占用或新增 GPU 资源大小
- Voxel / Volume Grid 显存占用
- 新增存储产物大小
- 运行时图片位置
- 光照效果图位置
- Ground Truth 图位置
- PSNR
- SSIM
- RMSE
- FLIP
- 误差结论
- 已知问题
- 是否达到本轮成功标准

## 8. 通过标准

每轮修改只有在满足以下条件时才可提交：

1. 程序能编译
2. 程序能启动
3. 能进入本轮目标涉及的模式或场景
4. 指标测试结果已写入 `docs/` 记录
5. 已明确写出本轮是否达到成功标准

如果未达到成功标准，也允许提交，但必须在记录中明确标注：

- 哪项失败
- 当前阻塞点
- 下一轮修复目标

## 9. 建议记录位置

建议采用以下结构：

- `docs/iteration-log-001.md`
- `docs/iteration-log-002.md`
- `docs/iteration-log-003.md`

每轮单独记录，避免不同轮次互相覆盖。
