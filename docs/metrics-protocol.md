# Volume Lighting 指标测试协议

## 1. 目的

本协议用于统一每轮修改完成后的测试指标、测试场景、记录格式和通过标准。

所有迭代在提交代码前，必须至少完成一次指标测试并写入对应迭代记录。

## 2. 指标来源与设计依据

本项目指标设计基于以下对标论文与当前核心算法目标：

- Clustered Deferred and Forward Shading
- Forward+
- Efficient Real-Time Shading with Many Lights

当前核心算法目标：

- 在 Clustered Light List 基线之上，引入单 Volume 单等效光源表示
- 对比性能收益与视觉误差
- 评估不同场景下的稳定性与画质损失

## 3. 固定测试场景

每轮测试默认使用以下场景：

1. `Sponza`
   用于性能、遮挡复杂度和分块规模验证

2. `MetalRoughSpheres`
   用于材质与高光误差验证

3. `DamagedHelmet`
   用于法线细节和近景高光误差验证

如果某轮改动只影响场景基础设施，可只测受影响场景；如果改动影响算法或 shader，必须覆盖全部三场景。

## 4. 固定对比模式

每轮测试至少包含以下模式：

1. `ClusteredBaseline`
   当前引擎原始 Clustered Light List 路径

2. `EquivalentVolumeV1`
   当前轮次的 Volume 等效光源路径

如果当前轮次尚未引入 `EquivalentVolumeV1`，则仍需记录 baseline 指标，作为后续对照起点。

## 5. 核心指标

### 5.1 性能指标

- 平均帧时间 `ms/frame`
- 平均帧率 `FPS`
- 光照 Pass GPU 时间
- 相对加速比
  计算方式：`Baseline 帧时间 / 当前方案帧时间`

### 5.2 画质指标

- 平均误差
- 峰值误差
- 高光区域误差
- 法线细节区域主观失真描述

说明：

- 在早期阶段若没有自动图像差分工具，可先采用“固定机位截图 + 主观误差记录”
- 当后续具备自动差分条件后，再补充定量误差脚本

### 5.3 稳定性指标

- 是否成功编译
- 是否成功启动
- 是否成功切换场景
- 是否成功切换渲染模式
- 是否出现闪烁、崩溃、黑屏或明显错误光照

## 6. 每轮必测项

### 6.1 文档或流程轮次

若本轮只改文档、流程或非渲染逻辑：

- 编译验证：可选
- 启动验证：至少一次
- 记录：说明“本轮无算法指标变化”

### 6.2 场景基础设施轮次

若本轮改动场景切换、场景配置或基础运行逻辑：

- 至少测试受影响场景
- 记录启动成功与否
- 记录截图是否正常

### 6.3 算法或 shader 轮次

若本轮改动算法、SSBO、聚合逻辑或 shader：

- 必测三场景
- 必测 baseline 与当前算法路径
- 必须记录性能指标
- 必须记录至少 1 组截图或主观误差结论

## 7. 固定记录字段

每轮测试记录必须包含：

- 轮次编号
- 日期
- 提交哈希
- 本轮目标
- 测试场景
- 测试模式
- 编译结果
- 启动结果
- 平均帧时间
- FPS
- 光照 Pass GPU 时间
- 截图位置
- 误差结论
- 已知问题
- 是否达到本轮成功标准

## 8. 通过标准

每轮修改只有在满足以下条件时才可提交：

1. 程序能编译
2. 程序能启动
3. 能进入本轮目标涉及的模式或场景
4. 指标测试结果已写入 `docs/` 记录
5. 已明确写出本轮是否达到成功标准

如果未达到成功标准，也允许提交，但必须在记录中明确标注：

- 哪项失败
- 当前阻塞点
- 下一轮修复目标

## 9. 建议记录位置

建议采用以下结构：

- `docs/iteration-log-001.md`
- `docs/iteration-log-002.md`
- `docs/iteration-log-003.md`

每轮单独记录，避免不同轮次互相覆盖。
