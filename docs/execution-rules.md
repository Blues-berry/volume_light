# Volume Lighting 执行守则

## 1. 目的

本守则用于约束后续所有迭代，防止需求膨胀、实现漂移和文档失配。

## 2. 基本原则

1. 先文档，后代码
   每轮实现前必须先核对需求文档、任务集文档和上一轮迭代记录。

2. 单轮单目标
   每轮只允许有一个主目标，例如“补实验场景”或“引入单等效光源数据结构”。

3. 不跨级实现
   当前目标若属于 `v1`，不得顺手实现 `v2` 内容。

4. baseline 永不删除
   新算法路径必须与 baseline 并存，直到实验结束。

5. 每轮必须可运行
   本轮结束时程序必须至少能编译、启动并进入当前目标路径。

6. 每轮必须可记录
   没有 `docs/` 记录的修改，不视为完成。

7. 每轮必须可回退
   提交按里程碑控制，禁止把多个实验方向揉进一次提交。

8. 超范围想法不直接实现
   任何不在当前文档范围内的新想法，先记录到延期事项，不直接进入代码。

9. 每轮必须测试
   每轮修改完成后，必须按 `docs/metrics-protocol.md` 执行指标测试并记录结果。

## 3. 当前默认范围

当前默认范围固定为 `EquivalentVolumeV1`：

- 单 Volume 单等效光源
- 三实验场景
- Baseline 与实验路径切换
- 基础截图、性能记录和误差观察

以下内容默认不进入当前轮次：

- 局部动态增量更新
- 双代表光源
- 方向基函数
- 间接光传播
- 稀疏体素结构

## 4. 执行前对齐规则

每轮执行前必须回答：

- 本轮目标是什么
- 对应需求文档的哪一节
- 本轮明确不做什么
- 成功标准是什么
- 需要更新哪些文档
- 完成后要跑哪些验证
- 本轮提交信息准备怎么写

如以上问题不能一次性回答清楚，则本轮不开始编码。

## 5. 文档职责划分

- 主需求文档：研究目标、当前范围、约束、成功标准
- 调研文档：论文与开源项目依据
- 任务集文档：阶段任务拆分与完成状态
- 迭代日志：每轮做了什么、结果如何、遗留什么
- 执行守则：如何开始与约束每一轮执行
- 指标协议：每轮需要测试哪些指标、如何记录、何时允许提交

## 6. Git 提交规则

- 每轮迭代结束前清理工作区，避免混入无关改动
- 每次提交必须对应单一主目标
- 每次提交应同时包含代码改动和对应 `docs/` 记录
- 每次提交前必须完成对应的指标测试记录
- 当前分支策略默认直接提交到 `main`
